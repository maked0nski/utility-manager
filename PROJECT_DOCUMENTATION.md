# UtilityManager: Повна технічна та продуктова документація

## 1. Мета проекту
`UtilityManager` автоматизує помісячний цикл обліку комунальних платежів та оренди для нерухомості:
- збір/введення показників,
- застосування тарифів,
- розрахунок нарахувань,
- облік оплат,
- контроль боргу/переплати,
- ведення історії орендарів і витрат власника.

Ключова бізнес-ціль: прибрати ручний Excel-облік як основне джерело правди. Excel використовується лише для одноразового імпорту історії.

## 2. Поточний стан реалізації (станом на зараз)

### Реалізовано
- Docker-інфраструктура (MySQL, API, Frontend, Adminer).
- Backend на FastAPI (Python 3.12), MySQL 8, SQLAlchemy ORM.
- Frontend на React + Vite (адмін-панель).
- JWT-авторизація адміна.
- Bootstrap-підказка дефолтного адміна (`admin/admin123`) + зміна пароля.
- Політика пароля (мінімальна складність) + рекомендація ротації пароля у `bootstrap-info`.
 - CRUD базових сутностей:
  - нерухомість,
  - орендар,
  - призначення орендаря (tenancy),
  - тарифи,
  - показники лічильників (включно з реєстрами `register_name`).

## 3. Бізнес-правила (узгоджена логіка)

## 3.1 Розрахунок комуналки
Для обраного місяця:
- `Нараховано` = сума всіх рядків таблиці розрахунку (включно з фіксованими та лічильниками).
- `Оплачено` = сума платежу(ів) за цей місяць (зараз зберігається як один актуальний запис per month).
- `Борг на зараз` = `Борг з минулого + Нараховано - Оплачено`.

## 3.2 Борг з минулого
- Переноситься лише з **підтверджених** (зафіксованих) місяців.
- Якщо місяць не підтверджений, його результат не впливає на `Борг з минулого` наступного місяця.
- Після підтвердження місяць все одно можна редагувати/перераховувати.

## 3.3 Оплати
- Комуналка: `POST /admin/payments/utilities` працює як `upsert` за `(apartment_id, year, month)`.
- Дата оплати (`paid_at`) прив’язана до конкретного місяця і повертається окремо API (`month_payment_date`).
- Допускаються переплати/недоплати: баланс переноситься далі зі знаком `+/-`.

## 3.4 Тарифи
- Тарифи версіонуються датою `effective_from`.
- Зміна тарифу у старому місяці застосовується з цього місяця далі до наступної зміни.
- Послуги можуть бути активні/неактивні з конкретного місяця.
- Для metered-послуг підтримується `initial_meter_reading`.

## 3.5 Відшкодування
- Відшкодування орендарю відображається у розрахунку мінусовим рядком.
- Зменшує суму до сплати за комуналку в межах місяця.

## 3.6 Оренда
- Ведеться окремо від комуналки:
  - нарахування оренди,
  - оплата оренди,
  - валюта,
  - підтвердження отримання.

## 8. UX-логіка адмінки (операційний сценарій)

## 8.1 Початкове налаштування
1. Створити нерухомість.
2. Створити/призначити орендаря (з дати старту).
3. Створити тарифи послуг (fixed/metered).
4. Для metered послуг задати стартовий показник.

## 8.2 Робота щомісяця
1. Обрати місяць.
2. Заповнити/оновити показники.
3. Перевірити тарифи, скоригувати за потреби.
4. Натиснути `Заповнити місяць послугами`.
5. Внести оплату комуналки (сума, дата, примітка).
6. За потреби внести відшкодування/витрати.
7. Після перевірки натиснути `Підтвердити нарахування`.

## 8.3 Історичні корекції
- Допускаються задні зміни тарифів/показників/оплат.
- Після змін виконується перерахунок у відповідних періодах.
- Для контрольованого бухгалтерського сценарію рекомендовано повторно перевіряти підтверджені місяці після корекцій.

## 11. Продуктивність і оптимізація (рекомендації на поточному етапі)

## 11.1 Backend
- Додати Alembic-міграції (замість неявних змін моделі).
- Індекси для гарячих запитів:
  - `Invoice(apartment_id, year, month)`
  - `UtilityPayment(apartment_id, year, month)`
  - `MeterReading(meter_id, year, month)`
  - `Tariff(apartment_id, service_name, effective_from)`
- Обмеження цілісності:
  - унікальність платежу комуналки на період (або офіційно підтримати many-payments).
- Пагінація для журналів витрат/ремонтів.
- Unit/integration тести на формули балансу і перенос боргу.

## 11.2 Frontend
- Продовжити винесення логіки з `App.tsx` у feature hooks/components (основна частина виконана; залишаються точкові локальні стани coordinator-рівня).
- Write-операції переведені на `useMutation` + `invalidateQueries`; далі потрібна лише уніфікація ключів інвалідації.
- Формальна валідація форм (RHF + Zod для основних модалок реалізована; далі розширити на всі форми).
- Зберігати user-preferences (сортування/фільтри) у `localStorage`.
- Додати e2e тести (Playwright).

## 11.3 Security
- Обмежити доступ до storage через перевірку ролей/URL-signing.
- Маскування чутливих полів у UI за замовчуванням.
- Регулярна ротація JWT secret та політика складності пароля.

## 16. Висновок

Проект на поточному етапі придатний для реальної щомісячної роботи по одному/кількох об’єктах із ручним введенням і контролем розрахунків. Найближчий фокус — стабілізація (тести/міграції/модульність), завершення фінансових документів (фактура/друк), і лише потім автоматизація зовнішніх джерел (парсери/OCR/бот).
