# Automation: Vodokanal IF / Показники + Тарифи

## 1. Ціль
Автоматизувати роботу з кабінетом `https://new.vodokanal.if.ua/kabinet-spozhyvacha/pereity-v-kabinet-spozhyvacha/` для:
1. Подачі показників лічильника у дозволене вікно дат через `Внести показники`.
2. Перевірки, що подані показники збережені у `Статистика показників`.
3. Синхронізації тарифів з кабінета для послуг:
   - `Водопостачання`
   - `Водовідведення`
   - `Абонентська плата (водоканал)`

## 2. Область даних і мапінг послуг
Мапінг назви картки в кабінеті до `service_name` у системі:
1. `Послуга водопостачання` -> `Водопостачання`
2. `Послуга водовідведення` -> `Водовідведення`
3. `Абонплата` -> `Абонентська плата (водоканал)`

Джерела в кабінеті:
1. Головна сторінка кабінета: картки послуг з блоком `Тариф: ...`.
2. `Внести показники`: форма внесення значень лічильника.
3. `Статистика показників`: журнал поданих значень і дат.

## 3. Бізнес-правила подачі показників
1. Подавати показники тільки якщо є поточні показники на вкладці `Розрахунок` у нашій системі.
2. Вікно подачі: з 25 числа до 3 числа включно (міжмісячне вікно).
3. Правило періоду подачі:
   - Період місяця `M` відкритий з `25` числа місяця `M` до `3` числа місяця `M+1`.
   - Приклад: для лютого 2026 вікно подачі: з **25.02.2026** до **03.03.2026** включно.
4. Якщо сьогодні поза вікном подачі -> статус `waiting` (не робити submit).
5. Якщо вікно підходить, але на `Розрахунок` немає валідного поточного значення -> статус `waiting`.
6. Якщо submit виконано, обов'язково перевірити запис у `Статистика показників`:
   - знайдений запис за цільовий період;
   - значення дорівнює відправленому;
   - дата запису у межах поточного запуску (або останній запис збігається з відправленим).
7. Перед submit обов'язково зробити pre-check у `Статистика показників`:
   - якщо запис з таким періодом і значенням вже існує, submit не виконувати (ідемпотентність).
8. Якщо верифікація не пройшла -> `error` з деталлю причини.
9. Повторний запуск у тому ж періоді має бути ідемпотентним:
   - якщо показник вже подано і підтверджено, повторно не відправляти.

## 4. Бізнес-правила синхронізації тарифів
1. Оновлювати тарифи з карток кабінета:
   - `Послуга водопостачання`
   - `Послуга водовідведення`
   - `Абонплата`
2. Парсити значення з рядка `Тариф: <value> ...` як десяткове число.
3. Правило порівняння:
   - `new <= current_db` -> `no_change`
   - `new > current_db` -> `updated` (оновити тариф)
4. Округлення тарифів: до 2 знаків після коми (`round(value, 2)`).
5. `effective_from`: перший день поточного місяця синхронізації (`YYYY-MM-01`), якщо інше не задано окремим правилом.
6. Для `Абонентська плата (водоканал)` застосовується той самий compare/update контракт, як для інших послуг.

## 5. Контракт станів UI
Мінімальні стани для обох частин (показники/тарифи):
1. `waiting` -> очікування вікна або даних.
2. `no_change` -> нове значення не більше поточного у БД.
3. `updated` -> дані успішно оновлені.
4. `error` -> помилка логіну, парсингу, submit або верифікації.

Tooltip (`auto_check_message`) має містити коротку причину:
1. Для показників: чому submit не виконано або як підтверджено submit.
2. Для тарифів: яке значення знайдено і яке правило спрацювало.

## 6. Контракт даних (ApartmentTariffSetting)
Для автоматизації використовуються поля:
1. `auto_check_enabled`
2. `auto_check_time`
3. `auto_check_timezone`
4. `auto_check_window_day_from`
5. `auto_check_window_day_to`
6. `auto_check_target_year`
7. `auto_check_target_month`
8. `auto_check_completed_for_period`
9. `auto_check_status`
10. `auto_check_message`
11. `auto_check_last_value_raw`
12. `auto_check_last_value_rounded`
13. `auto_check_last_checked_at`
14. `auto_check_last_updated_at`
15. `auto_check_next_at`

Рекомендовані дефолти для цього провайдера:
1. `auto_check_window_day_from=25`
2. `auto_check_window_day_to=3` (міжмісячне вікно)
3. `auto_check_timezone=Europe/Kyiv`
4. `auto_check_time=09:00`

## 7. Потік воркера
1. Вибрати активні налаштування для цільових послуг.
2. Перевірити планувальник (час/таймзона/вікно днів).
3. Логін у кабінет водоканалу.
4. Блок A (показники):
   - прочитати поточні значення з `Розрахунок`;
   - якщо вікно і дані валідні, відправити через `Внести показники`;
   - перевірити через `Статистика показників`;
   - записати статус.
5. Блок B (тарифи):
   - зчитати тарифи з 3 карток;
   - застосувати compare rule;
   - при необхідності оновити записи тарифів;
   - записати статус.
6. Оновити `next_at` та службові поля аудиту.

## 8. Безпека та надійність
1. Не логувати пароль/токени/сесію.
2. Парсинг робити стійким до пробілів, переносів, не-розривних пробілів.
3. При мережевій/кабінетній помилці встановлювати `error` і повторювати у наступний запуск.
4. Всі операції мають бути ідемпотентними для повторного запуску в той самий день.

## 9. Мінімальні тести
1. Дата поза вікном `25..3` -> `waiting`.
2. Дата у вікні, але немає поточних показників на `Розрахунок` -> `waiting`.
3. Дата у вікні + валідні показники -> submit -> запис є у `Статистика показників` -> `updated/no_change` за правилами.
4. Submit зроблено, але в статистиці немає відповідного запису -> `error`.
5. Тариф `new <= current_db` -> `no_change`.
6. Тариф `new > current_db` -> `updated` і новий запис тарифу з `effective_from=YYYY-MM-01`.

## 10. Сторінка "Автоматизації": налаштування і ручний запуск
На сторінці `Автоматизації` для цього провайдера мають бути доступні:

1. Глобальні параметри задачі:
   - `Увімкнено`
   - `Час запуску`
   - `Таймзона`
   - `Вікно днів` (`від=25`, `до=3`)
   - `Логін кабінета`
   - `Пароль кабінета` (masked + reveal toggle)

2. Керування напрямками:
   - `Подача показників` (toggle)
   - `Синхронізація тарифів` (toggle)
   - Селектор послуг для тарифів:
     - `Водопостачання`
     - `Водовідведення`
     - `Абонентська плата (водоканал)`

3. Ручний запуск:
   - Кнопка `Запустити зараз`.
   - Режими запуску:
     - `Повний` (показники + тарифи)
     - `Лише показники`
     - `Лише тарифи`
   - Ручний запуск обходить час планувальника, але не обходить бізнес-правила:
     - для показників правило вікна `25..3` залишається активним;
     - для тарифів compare/update правила залишаються активними.

4. Валідації форми:
   - `Від` і `До` мають бути валідними днями місяця (1..31).
   - Підтримати міжмісячне вікно (`from > to`) як нормальний кейс.
   - Без логіна/пароля запуск у `error` з читабельною причиною.

5. Відображення результатів:
   - `Останній запуск` (дата/час).
   - `Останній статус` (`waiting/no_change/updated/error`).
   - `Повідомлення` (`auto_check_message`).
   - `Наступний запуск` (`auto_check_next_at`).
   - Для ручного запуску показувати source=`manual`, для планового source=`scheduled`.

6. Ідемпотентність ручного запуску:
   - Якщо показники за період вже подані і підтверджені, повторний `Запустити зараз` не дублює submit.
   - Якщо тарифи не змінились, повторний запуск повертає `no_change` без нових записів тарифів.
